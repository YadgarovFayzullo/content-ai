# ИИ-агент: предложение действий («как в Notion»)

ИИ-менеджер канала теперь не только даёт фидбек по реальным данным, но и может
**предложить настроить расписание за владельца** и применить его в один клик после
подтверждения. Бэкенд готов; ниже — контракт для фронта (content-pilot).

## 1. Ответ ИИ может содержать предложение действия

`POST /api/admin/tenants/{tenant_id}/ai-chat` теперь возвращает дополнительное поле
`proposed_action` (помимо `reply`, `insights`, `session_id`):

```jsonc
{
  "reply": "…совет… Хотите, я настрою это расписание за вас?",
  "insights": { /* как раньше */ },
  "session_id": "…",
  "proposed_action": {            // null, если ИИ ничего не предлагает
    "type": "set_schedule",       // пока единственный тип
    "posts_per_day": 2,
    "daily": [                    // паттерн ОДНОГО дня (для превью в UI)
      { "time": "09:00", "content_type": "repost" },
      { "time": "19:00", "content_type": "topic" }
    ],
    "slots": [ /* 7×daily — готовая недельная сетка для apply, weekday 0..6 */ ],
    "timezone": "Asia/Tashkent", // TZ, в которой заданы времена (TZ расписания)
    "peak_hour_local": 19         // час пика активности (или null)
  }
}
```

- Времена считает **бэкенд** из реального пика активности и в TZ расписания бота —
  фронт их не придумывает и не конвертирует. `daily` — для красивого превью
  («каждый день: 09:00 репост, 19:00 тема»). Тип `content_type`: `repost` | `topic`.
- Объём уже зажат лимитом тарифа (`max_posts_per_day`), а repost-слоты предлагаются
  только если тариф разрешает `repost_mode`.

### UI

Когда `proposed_action != null`, под ответом ИИ показать карточку-подтверждение с
превью из `daily` и кнопками **«Да, настроить»** / «Отмена». Текст-предложение уже
есть в `reply` (на языке канала).

## 2. Применение по подтверждению

При нажатии «Да, настроить»:

```
POST /api/admin/tenants/{tenant_id}/ai-chat/apply-action
{
  "slots": <proposed_action.slots>,   // echo обратно как есть
  "session_id": "<session_id>"        // опционально — залогировать подтверждение в беседу
}
```

Ответ:

```jsonc
{
  "applied": true,
  "confirmation": "Готово ✅ Настроил автопостинг: каждый день 2 пост(ов) — 09:00 — репост, 19:00 — тема. …",
  "schedule": { "slots": [ /* сохранённая недельная сетка */ ] }
}
```

- Эндпоинт **повторно валидирует** формат и тариф (тот же гейтинг, что у `PUT
  /schedule`) — подделать payload и обойти лимиты нельзя. При нарушении тарифа —
  402/403 с тем же телом ошибки, что и обычное обновление расписания.
- Если передан `session_id`, строка `confirmation` уже добавлена ассистентом в
  беседу — фронт может либо показать её из ответа, либо перечитать беседу.
- После успеха обновить вкладку «Расписание» (`GET /schedule`) и календарь —
  сетка теперь `schedule_mode="weekly"`.

## Замечания

- `apply-action` сейчас эквивалентен `PUT /schedule` плюс лог в чат, поэтому
  отдельного права не вводит: владелец и так может менять своё расписание.
- Расширение на другие действия (сменить `content_mode`, добавить источник и т.п.)
  идёт по той же схеме: новый `type` в `proposed_action` + ветка в apply-эндпоинте.
